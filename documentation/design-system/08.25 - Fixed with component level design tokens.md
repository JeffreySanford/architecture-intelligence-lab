# 08.25 - Fixed with component-level design tokens

## Overview

This story documents a lab-focused demo for PrimeNG styling recovery when Angular `ViewEncapsulation.None` causes component styles to behave globally.

The key message is:

- Production architecture: the application or enterprise design-system foundation owns design tokens.
- Lab story: Tab Four defines a localized token boundary to demonstrate how token-based recovery can restore predictable PrimeNG styling after encapsulation leakage.

This tab is intentionally self-contained for inspection and comparison. It should be framed as a localized proof rather than the final enterprise token source of truth.

## Tab title

**Tab 4: Encapsulation.None Token Recovery**

## Story

This tab intentionally uses `ViewEncapsulation.None` to simulate a global-style leakage scenario while reproducing the full Normal PrimeNG component set. The tab shows the same shared PrimeNG suite plus the page preview controls from the normal baseline, and it demonstrates how a scoped token boundary can recover predictable styling by mapping local design tokens into PrimeNG CSS variables.

> These design tokens are scoped to the local `.token-recovery-demo` wrapper, so they do not leak beyond this component’s DOM subtree.

The core story should be:

1. Angular component uses `ViewEncapsulation.None`.
2. Local component styles become global.
3. Broad `.p-*` overrides create style conflicts.
4. PrimeNG preset/design tokens become the safer system-wide fix.
5. Scoped tokens handle local exceptions.

## Token-framework explanation

PrimeNG supports a base + preset architecture:

- Base CSS uses CSS-variable placeholders.
- Preset supplies design token values.
- Tokens are organized into primitive, semantic, and component tiers.
- PrimeNG recommends design tokens over direct `.p-*` overrides.

For this tab, the demo uses:

- local CSS custom properties scoped under `.token-recovery-demo`
- a local mapping into selected PrimeNG `--p-*` variables
- a recovery-focused layout that reproduces the Normal PrimeNG component set

## Important distinction

Use these levels:

1. Global PrimeNG preset tokens — best for brand, surfaces, focus, inputs, buttons, cards.
2. App bridge tokens — CSS variables that map custom app layout to PrimeNG variables.
3. Scoped `dt` tokens or local custom properties — one-off exceptions for a specific surface.

## Local containment

This tab demonstrates a local token boundary inside the `Encapsulation.None Token Recovery` component. The recovery styling is intentionally scoped to the `.token-recovery-demo` wrapper, so the token values and PrimeNG variable mappings do not leak into other application areas.

Because the CSS custom properties are defined on a local wrapper and the PrimeNG bridge variables are set only inside that wrapper, the recovered styling is contained within this tab. Other pages or components using different styling will continue to render with their own styles.

This is why the story is framed as a localized demo boundary: it is a safe, one-off recovery surface, not a new global theming layer.

## Demo guidance

- Use local tokens and PrimeNG variable mappings in the recovery layout.
- Do not claim that the local tab tokens are the final enterprise design system.
- Explicitly state that production tokens belong at the app/design-system foundation.
- Avoid `::ng-deep`.
- Avoid broad unscoped `.p-*` rules outside the demo wrapper.
- Avoid using `!important` except for the intentionally simulated problem side.
